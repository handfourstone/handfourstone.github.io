IPv6

简介

IPv4 和 IPv6 之间基本的工作原理是不同的。和 IPv4 相比较，IPv6 有许多值得肯定（remarkable）变化。IPv6 中一个主要的修正（revision）是引入了扩展头。不像 IPv4 的选项那样，IPv6 中的每个扩展头都可以单独的被上层协议识别，从而扩大了选项部分的大小和数目。

其它类型的扩展头信息同样基于扩展头框架的方式得到应用。例如，IPv6 中包分段（packet fragmentation）和源路由（source routing）都采用扩展头的方式。由于引进的扩展头的概念，所以 IPv6 的头部被简化到只包含一些基本信息（essential information）和共有的操作（common operation）。

IPv6 另外一个重要的改变是出口封包的源地址选择。就像第二章介绍的那样，IPv6 提供了多种地址类型，不同类型的地址有着不同的属性：不同的地址范围或者不同的使用期（lifetimes）。对于一个给定的目的地址，根据优先级（preference）原则或者最优化（optimization）原则选择（某些情况下它俩是相互牵连的），选择一个合适的源地址已经不是什么难题了。

本章，我们将详细讨论 IPv6 在网络层上进行操作的具体问题。首先，根据 [RFC2460] 我们将提过一些关于 IPv6 的扩展头的概述（overview）。然后，根据 [RFC3484] 我们将讨论一些默认的地址选择算法。虽然本章的重点在介绍源地址选择算法上，但是我们也会介绍一些关于目的地址选择的算法，因为这些算法有着相同的基本规则。

IPv6 头部格式

任何 IPv6 的封包都会包含一个固定大小的头部，这个头部叫做 IPv6 头部，IPv6 头部大小是 40 个字节，里面包含传送传送一个 IPv6 封包的最基本的字段。
表 3-1 给出了 IPv6 头部结构。

Version 版本字段，IPv6 包中，该字段为 6 。
Traffic Class 传输等级，8 比特，该字段意味着末节点会生成一个新的传输等级（优先级），然后让中间的节点根据每个包中的传输等级来转发这些封包。源节点默认会将该字段设置为 0 。无论该字段最初是否设置为 0 ，在封包到达目的地的路由过程中都可能会发生改变。[RFC2474] 和 [RFC3168] 定义了该字段的使用细节，这些内容已经超出了本书的范围。
Flow Label 流标签，20 比特，该字段提供了一种区分不同传输流（traffic flow）的手段。[RFC3697] 给节点处理该字段提供了一些指导：分包的源节点确认了一个分开的流，例如，一个具体应用的数据流有一个非零的流标签；如果源节点不想让数据包和传输流控制有联系，那么源节点就必须将该字段设置为 0 ；流标签的值在封包到达目的地的路由过程中是不能改变的。需要特别说明的是，存在一个如何在源节点到目的节点之间（包括中间的路由节点）建立一个流状态的开放问题（open issue）。
Payload Length 该字段是一个 16 比特的无符号整型数据，值等于封包的大小减去 IPv6 头部的大小。也就是说，如果有扩展头的话，该值等于扩展头的大小与上层协议数据大小之和。
Next Header 8 比特的一个数据表明 IPv6 头部之后的协议类型或者是扩展头的类型。
Hop Limit 8 比特的一个数据表明一个封包到达目的地之前可以经过路由转发的次数。每个路由在转发该封包的时候都会对该字段减一，如果该字段为 0 就会丢弃该封包。
Source Address 128 比特的字段存储封包的源 IPv6 地址。
Destination Address 128 比特的字段存储封包的目的 IPv6 地址。如果路由头命中的话，目的地址就可以是中间节点的地址，否则目的地址就是封包里面的目的地址。

IPv4 和 IPv6 头部比较

IPv6 扩展头

3.2.1 节中介绍过，IPv4 中选项的概念在 IPv6 中体现在扩展头中。每个扩展头中表明一些具体的选项特性，或者一些具体的工作，例如指明高层的协议。因为路由器不会去处理扩展头的， 除非被明确要求处理，所以 IPv6 在转发有着端到端特性的封包的时候就可以变得更轻量级。同样，扩展头长度可以比 IPv4 选项部分的最大长度还要大很多。

[RFC2460] 定义了下列一些扩展头：一跳一跳选项头（Hop-by-Hop option header），路由头（Routing header），分段头（Fragment header），目的选项头（Destination option header）。此外， IP 安全协议（IPsec）定义了两个额外的扩展头：认证头（Authentication header）和封包安全负载头（Encapsulating Security Payload header）[RFC4302][RFC4303]。扩展头位于 IPv6 头部和高层协议头之间。扩展头在它被放入封包的时候处理。一般来说，高层协议就是扩展头链（多个扩展头）的终结。在没有高层协议的封包中会使用一个虚拟的头（No Next header）表明扩展头链的结束。

每个扩展头的大小都是 8 字节的整数倍（除了认证头，它的大小是 4 字节的整数倍）。扩展头的大小不包含前 8 个字节，因为所有的扩展头的大小都大于 8 字节。扩展头内的每个字段都会对齐到它数据部分的自然边界上面，如果有必要的话就会进行填充处理。

其他用来扩展 IPv6 功能的规格中可能也定义了一些扩展头，例如，移动设备（Mobile）IPv6 有它自己的扩展头，这个将在 IPv6 高级协议实现 一节中进行讨论。

扩展头的顺序

当一个包里面有若干个扩展头的时候我们会有一个推荐的顺序。表 3-2 展示的就是推荐的顺序。注意的是这个顺序并不是强制要求的，发送节点需要遵循这个顺序，但是接受节点应该假设发送节点不一定是按照这个推荐顺序来组合扩展头的。Hop-by-Hop options header 是一个例外，这个扩展头必须紧跟在 IPv6 的头后面，而且一个封包中这个扩展头只能出现一次。

推荐里面声明任何一个扩展头（除了 Destination options header）最多只能出现一次。Destination options header 可以出现两次，一次出现在 Routing header 之前，一次出现在 Routing header 之后。出现在 Routing header 之前的 Destinating options header 可以被列在 Routing header 中的所有中间路由节点处理。出现在 Routing header 之后的 Destination options header 只能被封包的接收者处理。同样，移动（Mobile） IPv6 指出另外一个 Destination options header 的位置，该位置位于 Routing header 和 Fragment header 之间，关于 Mobile IPv6 将在第五章和 IPv6 高级协议实现 中讨论。

扩展头的顺序仅仅是一个建议，这就意味着 IPv6 的实现要能处理以任何顺序排放的多个扩展头（任意一个扩展头在一个封包中都可能多次出现）。

Hop-by-Hop options header

Hop-by-Hop options header 会被路由过程中的每个节点检查和处理，例如，例如负责将封包转发到目的地址的中间路由器。当然，源节点和目的节点也会处理 Hop-by-Hop options header。在 Hop-by-Hop 之前一个一个头中的 next_header 字段会被设置成 0。Hop-by-Hop 字段在 IPv6 中被要求紧跟在 IPv6 的头之后。图 3-3 展示了Hop-by-Hop options header 扩展头的格式。

Routing header

源节点使用 Routing header 来列出封包在传输路径上的中间节点（称作路由分段）Routing header 之前的节点中的 Next Header 字段值为 43。图 3-5 给出了 Routing header 的一般格式。


图 3-5 Routing header 格式

Next Header 8 比特的字段，值为协议号，表明紧跟在 Hop-by-Hop options header 之后的协议头。
Hdr Ext Len 8 b比特的字段，值为协议头的长度，单位是 8 字节，注意不包括 Next Header 这 8 个字节。
Routing Type 8 比特的字段，值为 Routing header 的类型码。IPv6 可以有多种类型的 Routing header。[RFC2460] 定义了类型为 0 的 Routing header，这个将在下面进行讨论。
Segments Left 8 比特的字段，指明路由分段数，也就是说封包在到达目的节点之前还需要进行访问的中间节点数。
Type Specific data 根据相应的 Routing Type，该字段包含相应的数据。

如果 Segment Left 字段为 0 的话，正在接受封包的节点将会忽略一个无法识别的 Routing header（换句话说，也就是一个未知的 Routing Type），在这种情况下，节点将继续处理无法识别的 Routing header 之后的节点。如果 Segment Left 如果是一个非零值，那么节点将会丢弃这个封包，并且生成一个 ICMPv6 的报文发送给源节点来通知这种“参数错误”的情况。这个错误的消息将会在 4.2.4 节中讨论。
